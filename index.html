01

Project Overview

UTM Skudai's campus shuttle service currently operates on fixed loops with fixed schedules. Buses complete the same route regardless of actual passenger demand — causing overcrowding at peak stops while other buses run nearly empty. This is a coordination failure: the information exists (students are queuing), but no mechanism acts on it intelligently.

This project proposes a Multi-Agent System (MAS) that transforms each bus and each bus stop into an autonomous agent. Stop Agents monitor queue levels and broadcast demand signals. Bus Agents receive those signals, evaluate whether rerouting serves more passengers efficiently, and update their routes dynamically — coordinating through a message broker without any central controller.

Scope Clarification

This is a simulation system, not a live hardware deployment. Queue counts, bus positions, and passenger loads are all generated by a demand simulation engine — not measured from physical sensors. This is academically standard, and is consistent with how real production systems work: Skyss (Norway) and Moovit (Italy) both estimate stop demand indirectly through onboard boarding/alighting data rather than direct headcounts at stops. Your simulation applies the same indirect inference principle, using UTM's class schedule as the pattern source instead of historical APC sensor logs.

01.5

Real-World Grounding

A common question about this project is: "If you can't measure the queue directly, is the simulation realistic?" The answer is yes — and it's backed by how production transit systems actually work.

Skyss — Norway

Skyss (the public transit authority for Vestland county) estimates waiting passengers at stops indirectly through Automatic Passenger Counting (APC) data from onboard door sensors. By analysing boarding and alighting patterns across multiple trips — higher boardings during peak hours, consistent net inflows at certain stops — they model expected queue demand without any sensor physically at the stop. Sensor accuracy is 90–95%. This feeds real-time apps alerting users to wait times and crowded stops.

Moovit — Italy

Moovit infers stop crowding by combining APC onboard data with crowdsourced user reports. If APC shows heavy boardings relative to bus capacity, the system flags potential queuing. Algorithms weigh recent data in 10–30 minute rolling windows against schedule patterns to estimate current waiters — displaying levels like "uncrowded" or "full" to guide riders.

Why This Matters for This Project

Both Skyss and Moovit confirm that direct headcounting at bus stops is neither standard nor necessary in production systems. Demand is always inferred from boarding/alighting patterns — the same principle your demand simulation engine applies. The only difference: Skyss uses years of real APC sensor logs; this project substitutes those with UTM's known class schedule patterns. This is a legitimate and directly citable justification for the simulation approach in the Literature Review.

Japan — Stop-Event Positioning

Several Japanese city bus networks (Toyama, parts of Osaka feeder routes) use RFID transponders on buses with fixed readers at stops. When a bus passes, the reader logs: Bus 7 passed Stop 14 at 09:23. No GPS required — position is inferred from stop-event transitions. This is the real-world basis for this project's bus positioning model, where location is expressed as "last confirmed stop + ETA to next stop" rather than continuous coordinates.

03

Problem Statement

The current UTM Skudai campus bus service operates on a static, schedule-driven model that is structurally unable to respond to real-time demand variation. Three specific problems arise from this:

P1 — Demand-Supply Mismatch

Bus routes and headways are fixed regardless of how many students are actually waiting. During pre-lecture periods, certain stops (Arked Cengal, PAP Indah) accumulate queues of 30–40 students while buses on other routes carry fewer than 10 passengers.

P2 — No Coordination Between Buses

Multiple buses operating simultaneously have no mechanism to communicate or avoid redundant coverage. Two buses can arrive at the same stop consecutively while another stop goes unserved for an extended period.

P3 — No Feedback Loop

Fleet managers have no visibility into real-time occupancy, queue distribution, or route efficiency. Decisions about route adjustments are made manually, reactively, and infrequently.

Why Existing Solutions Don't Solve This

Simple GPS trackers show where buses are but provide no demand intelligence. Centralised dispatch systems solve P2 but introduce a single point of failure and do not scale well. Standard scheduling software optimises for consistency, not responsiveness. A Multi-Agent System addresses all three problems through local agent reasoning producing globally adaptive behaviour.

Objectives

  1. To design and implement a Multi-Agent System in which bus and stop entities operate as autonomous agents capable of local decision-making without centralised control.
  2. To model UTM Skudai's campus road network as a directed weighted graph and implement Dijkstra's algorithm for real-time route utility calculation by Bus Agents.
  3. To develop a demand simulation engine that generates realistic queue and boarding data based on UTM's class schedule patterns, serving as the system's primary input source.
  4. To implement an MQTT-based inter-agent communication protocol enabling Stop Agents to broadcast demand states and Bus Agents to coordinate task allocation without double-response conflicts.
  5. To build a real-time web dashboard that visualises agent positions, states, route decisions, and system logs for fleet management use.
  6. To validate system correctness and functional behaviour under simulated demand scenarios, using black box test cases and defined software acceptance criteria to confirm the system meets its specified requirements.
⚠️ System Development Track — Critical Framing Note

Objective 6 is deliberately worded around software validation, not algorithmic comparison. Framing evaluation as "MAS vs fixed-route to prove performance" risks the panel reclassifying this as a Research-based project (which requires CGPA 3.30 minimum). The comparison run exists as a performance acceptance test — one test case among many confirming the software meets its requirements — not as a research contribution. Always frame testing and results in terms of whether the system meets defined acceptance criteria.

Layer 1 — Graph Engine (Campus Digital Twin)

UTM Skudai's road network is represented as a directed weighted graph. One-way roads are modelled as directed edges. Edge weights are dynamically updated by the formula:

W = W_distance × (1 + 1 / (Queue_count + 1))

A higher queue at the destination lowers the edge weight penalty (making it more attractive to route toward). Distance remains the base weight — the formula balances urgency against travel cost.

Named stops in the graph (based on actual UTM Skudai routes):

  • Arked Cengal (highest demand during AM peak)
  • Arked Angkasa
  • PAP Indah
  • Kolej Tun Razak (KTR)
  • Kolej Datin Seri Endon (KDSE)
  • Kolej Tun Fatimah (KTF)
  • Fakulti Komputeran (FC)
  • Fakulti Kejuruteraan (FK)
  • Bangunan Canselori / Admin Hub

Layer 2 — Agent Layer

See Section 06 (Agent Design) for full detail. All agents run as concurrent processes, each maintaining their own state and communicating only via the MQTT broker — never directly calling each other.

Layer 3 — MQTT Communication Layer

A Mosquitto MQTT broker (running locally) handles all inter-agent messaging. Topic structure:

A web interface (React or plain HTML/JS with Leaflet.js map) showing: live bus positions on a UTM campus map, stop queue heatmap (green/yellow/red), per-bus state and load bar, agent decision log feed, and simulation controls (speed, scenario selector, manual override).

06

Agent Design

Stop Agent
Reactive Agent · One per Bus Stop
  • Receives queue count from demand simulation engine every 5 seconds
  • Classifies queue into LOW / MEDIUM / HIGH / CRITICAL thresholds
  • Broadcasts demand state to MQTT topic stops/{id}/state
  • Monitors whether a Bus Agent has claimed the stop and suppresses re-broadcast if already claimed
  • Resets claim status after bus departs and queue drops below threshold
Bus Agent
Deliberative Agent · One per Bus
  • Maintains state machine: Idle → Commuting → Boarding → Recalculating
  • Subscribes to all Stop Agent broadcasts on MQTT
  • On receiving HIGH/CRITICAL signal: runs Dijkstra to calculate Utility Score for diverting
  • Checks capacity constraint before accepting divert (will not reroute if load ≥ 85%)
  • Checks protected time zone (no rerouting within 12 min of any lecture slot)
  • Checks journey delay limit (will not reroute if detour adds more than 10 min to any current passenger's remaining journey)
  • Publishes claim lock if diverting; ignores broadcast if another Bus Agent already claimed
  • Reverts to default route if MQTT broker is unavailable (graceful degradation)
Demand Simulation Engine
Environment Model · Not an Agent
  • Generates queue counts per stop based on time-of-day and UTM class schedule
  • Models pre-lecture peaks (8:45, 9:45, 11:45, 12:45, 2:45pm) as demand spikes
  • Tracks rolling boarding/alighting history per stop (mirrors Skyss APC indirect inference approach)
  • Adjusts demand estimates based on recent net inflow patterns at each stop
  • Simulates infrared counter logic: net passengers = board − alight per stop
  • Configurable: noise factor, peak intensity, scenario presets
Dashboard Controller
Observer / Admin Interface
  • Subscribes to all MQTT topics in read-only mode
  • Renders bus positions, routes, and stop states in real-time
  • Displays agent decision log with timestamps and reasoning
  • Allows admin to manually override a bus route (publish to override topic)
  • Provides simulation controls: play, pause, speed multiplier, scenario select
  • Shows comparison metrics: MAS mode vs Fixed Route mode side-by-side

Execution Scenario (Walkthrough)

The following traces a single demand-response event as it flows through the full system at 8:47am on a Tuesday:

1
Demand engine generates queue = 38 at Arked Cengal (pre-9am lecture peak). Stop Agent classifies this as CRITICAL.
2
Stop Agent publishes to stops/arked_cengal/state: { queue: 38, priority: CRITICAL, claimed: false }
3
Bus Agent 1 (near KDSE, load: 6/28, state: Commuting) receives the broadcast. Checks protected time zone: 8:46am is outside the 12-min pre-lecture window (next slot: 9:00am, still 14 min away — clears). Runs Dijkstra: divert adds 2.3 min to route. Journey delay check: 2.3 min is well under the 10-min threshold for all 6 current passengers. Utility Score: HIGH. Decides to divert.
4
Bus Agent 1 publishes claim to tasks/arked_cengal/claim. Transitions state to Recalculating, updates Route_Vector, transitions back to Commuting on new route.
5
Bus Agent 2 (near FK, load: 24/28, state: Commuting) also received the broadcast. Checks capacity: load ≥ 85%. Constraint fails. Ignores broadcast. Continues current route.
6
Stop Agent receives claim from Bus 1. Sets claimed: true, suppresses further broadcasts for this demand event.
7
Bus 1 arrives at Arked Cengal 5.8 min later. Boarding simulation: 28 passengers board (fills bus). Alighting simulation: 6 alight at Cengal. Net load: 28/28 (full).
8
Dashboard logs: "08:47:12 — Bus 1 diverted to Arked Cengal. Reason: Queue CRITICAL (38). Utility score: 0.84. Constraint checks passed." Queue wait time for those 38 students: ~6 min vs ~22 min baseline.
08

Design Constraints & Rules

These rules are hard-coded into Bus Agent logic to prevent the system from optimising itself into a user-hostile state:

ConstraintRuleRationale
Minimum Service Guarantee Every stop must receive a bus visit at least once every 20 minutes, regardless of demand signals. Prevents low-demand stops from being permanently ignored by the optimisation logic.
Capacity Lock Bus Agent will not accept a divert if current load ≥ 85% of max capacity (≥24/28). Prevents routing a full bus to a high-demand stop, which helps no one.
Protected Time Zone No rerouting within 12 minutes of any standard lecture slot (8am, 9am, 10am, 11am, 12pm, 2pm–5pm). Global guard — applies to all buses. Coarse protection near lecture boundaries without needing to track individual student schedules.
Journey Delay Limit Bus Agent will not divert if doing so adds more than 10 minutes to any current passenger's remaining journey to their destination stop. Configurable threshold. Respects all passengers regardless of their reason for travelling — early arrival, delayed class, or otherwise. Does not assume class times.
Claim Lock (Anti Double-Response) Once one Bus Agent claims a high-demand stop, all other agents suppress their response to that same event. Prevents two buses racing to the same stop, leaving other stops unserved.
Directed Graph Compliance Route recalculations must comply with the directed graph — one-way roads cannot be traversed in reverse. Reflects UTM's actual one-way road segments (e.g., the road past Arked Angkasa).
Graceful Degradation If the MQTT broker is unreachable for >10 seconds, all Bus Agents revert to their default fixed route. Ensures the system fails safely — no bus is left without instructions.
Known Unresolved Tradeoff

Dynamic routing cannot fully guarantee consistent journey times for through-passengers. A student boarding at KTR heading to FC may experience a slightly longer journey if their bus diverts en route. The Journey Delay Limit (10 min threshold) and Protected Time Zone constraints reduce this risk significantly but do not eliminate it entirely — this is acknowledged as a known system limitation.

Functional Requirements

FR1 — Campus Graph

  • System shall represent UTM Skudai bus network as a directed weighted graph with named nodes and edges.
  • Edge weights shall be dynamically recalculated based on queue data from Stop Agents.
  • Graph shall support Dijkstra's shortest/optimal path query per Bus Agent request.

FR2 — Stop Agent Behaviour

  • Each Stop Agent shall receive queue count input every simulation tick (5 seconds).
  • Each Stop Agent shall classify queue level into thresholds: LOW (<5), MEDIUM (5–15), HIGH (16–30), CRITICAL (>30).
  • Stop Agent shall publish demand state to MQTT only when status changes or at 30-second intervals.
  • Stop Agent shall track claim status and suppress redundant broadcasts.

FR3 — Bus Agent Behaviour

  • Bus Agent shall maintain and publish its state, position (last stop + ETA), load count, and route at all times.
  • Bus Agent shall evaluate rerouting utility on receipt of HIGH or CRITICAL stop broadcasts.
  • Bus Agent shall check all constraints (capacity lock, protected time zone, journey delay limit, claim lock) before accepting any divert.
  • Bus Agent shall publish a claim lock before diverting and release it after completing the stop visit.
  • Bus Agent shall log all decisions (reroute accepted, reroute rejected, reason) to the system log topic.

FR4 — Dashboard

  • Dashboard shall display all bus positions on a schematic UTM campus map updated every simulation tick.
  • Dashboard shall display per-stop queue level with colour coding.
  • Dashboard shall show a scrollable decision log with timestamps and reasoning.
  • Dashboard shall provide simulation controls: play/pause, speed (1×, 5×, 10×), scenario preset selector.
  • Dashboard shall display system performance acceptance test results: MAS mode metrics against defined acceptance criteria targets, with a fixed-route reference run for context.
  • Admin user shall be able to manually override any bus's current route via the dashboard.

FR5 — Demand Simulation Engine

  • Engine shall generate queue counts per stop based on configurable time-of-day demand profiles.
  • Engine shall model lecture-time demand spikes at defined time slots.
  • Engine shall simulate boarding and alighting volumes per bus stop visit.
  • Engine shall support scenario presets: Normal Day, Peak Morning, Exam Week, Low Demand.

Non-Functional Requirements

CategoryRequirement
Performance Agent decision cycle (receive broadcast → evaluate → publish response) shall complete within 500ms per agent. Dijkstra computation on the campus graph (≤15 nodes) shall complete in under 50ms.
Reliability System shall enter graceful degradation (fixed-route fallback) within 10 seconds of MQTT broker loss. No agent shall enter an undefined state under any simulated input combination.
Scalability Adding a new bus or stop shall require only adding a new agent instance with configuration — no changes to core coordination logic.
Usability Dashboard shall be operable by a non-technical fleet manager with no training beyond a 5-minute orientation. Driver route display shall show only next waypoint and direction — no complex UI.
Maintainability Agent logic and graph configuration shall be separated into distinct modules. Demand profiles shall be editable via configuration file without code changes.
Testability All agent decision logic shall be unit-testable with deterministic inputs. System shall support a headless simulation mode (no UI) for automated test runs.

Technology Stack

Budget: Zero

All technologies selected are free and open-source. No LLM APIs, no cloud hosting costs, no paid services required. Everything runs locally on a development machine.

ComponentTechnologyJustification
Agent Runtime Python 3.x Strong MQTT library support (paho-mqtt), clean multiprocessing/threading for concurrent agents, easy graph library integration.
Graph & Pathfinding NetworkX (Python) Built-in Dijkstra implementation, directed graph support, lightweight. No external dependency cost.
Message Broker Eclipse Mosquitto (MQTT) Industry-standard, free, runs locally. Lightweight enough for simulation scale. Well-documented.
Dashboard Frontend React + Leaflet.js or plain HTML/JS Leaflet provides free campus map rendering. React state management suits real-time updates. Plain HTML/JS is fallback if complexity is a concern.
Dashboard Backend FastAPI (Python) + WebSockets Lightweight, fast to set up, native WebSocket support for pushing real-time agent states to the browser.
Data / Config JSON files Graph topology, stop coordinates, demand profiles, and agent configurations stored as JSON — human-readable, version-controllable, no database needed.
Testing pytest Unit tests for agent decision logic, graph calculations, and constraint checks.
On LangGraph / LLMs

LangGraph is designed for orchestrating LLM-based agents and is not appropriate here. Bus Agents are rule-based state machines with deterministic logic — LLMs would add cost, non-determinism, and latency without any benefit. If a natural language query interface ("which bus is closest to FC?") is desired as an optional add-on, a lightweight LLM API call could be added in the final weeks of development only if time permits.

Scope & Limitations

In Scope

  • Full MAS implementation: Stop Agents, Bus Agents, MQTT coordination layer
  • Directed weighted campus graph of UTM Skudai shuttle network
  • Demand simulation engine with time-aware queue generation
  • Real-time web dashboard with map visualisation and decision logs
  • Software acceptance test under simulated demand (performance against defined criteria, with fixed-route reference run)
  • Admin manual override functionality
  • Unit tests for agent decision logic and constraint rules

Out of Scope (This Semester)

  • Live hardware deployment on actual UTM buses
  • Real-time GPS tracking or physical RFID sensors
  • Student-facing mobile app (dashboard is fleet-manager only)
  • Integration with UTM's existing IT or transportation systems
  • Multi-route optimisation across more than one bus line simultaneously (single corridor simulation)

Acknowledged Limitations

  • Queue data is simulated, not measured — real deployment would require physical counters or camera-based detection at stops
  • Bus position uses stop-event-based tracking (last confirmed stop + ETA), not continuous GPS coordinates
  • Time buffer constraint reduces but does not eliminate the risk of through-passengers experiencing longer journeys during reroutes
  • Demand model is calibrated to general UTM patterns, not validated against actual ridership data
Real-World Deployment Path — System Maturity

The simulation proves the agent coordination logic first. Physical deployment builds on top of it in stages — the only hardware needed to start is a cheap infrared door counter on each bus.

Four Phases of Maturity

PhaseWhat ChangesDemand Source
Phase 1 — PSM1 (Now) Simulation only. Agent logic proven. No hardware. Class schedule model
Phase 2 — Early Deployment Infrared door counters fitted on buses. Real boarding/alighting data starts accumulating. Schedule model + real boarding logs begin building
Phase 3 — Pattern Refinement After 1–2 semesters of real data, system overrides schedule where real patterns differ. Rainy days, exam weeks, Friday slots all detected automatically. Boarding history overrides schedule where they diverge
Phase 4 — Skyss Level Boarding history alone predicts demand accurately. Schedule becomes fallback only. Prediction improves every semester automatically. Boarding history (APC equivalent) — schedule as fallback
Why the Door Counter Is the Only Critical Hardware

It is cheap, privacy-safe (counts people, doesn't identify them), mounts at the bus door with no stop infrastructure needed, and produces exactly the boarding/alighting data that feeds the improvement loop — the same data Skyss (Norway), NUS/NTU Singapore, and Seoul campus shuttles rely on. Every semester of real operation makes the demand model more accurate without any code changes.
